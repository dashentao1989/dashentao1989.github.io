git fetch与git pull的区别

发表于 2016-08-11   |  

git fetch

场景：本地分支拉取远程分支最新commit数据到本地。

操作：

  • git fetch origin
  • git log -p ..FETCH_HEAD
  • git merge FETCH_HEAD

经过上面的操作分析下FETCH_HEAD关键词：

  • FETCH_HEAD

FETCH_HEAD is a short-lived ref, to keep track of what has just been fetched from the remote repository. git pull first invokes git fetch, in normal cases fetching a branch from the remote; FETCH_HEAD points to the tip of this branch (it stores the SHA1 of the commit, just as branches do). git pull then invokes git merge, merging FETCH_HEAD into the current branch.
The result is exactly what you’d expect: the commit at the tip of the appropriate remote branch is merged into the commit at the tip of your current branch.

说明如下：

FETCH HEAD是一个暂时的参照区，作用是对从远程仓库拉下的最新代码保持追踪，git pull首先会调用git fetch, 正常情况下从远程基础上拉取一个分支，FETCH_HEAD指向该分支的顶部（他存储了SHA1的提交，也就是远程分支之前所提交的一系列新的commit操作），git pull然后调用git merge操作， 合并FETCH_HEAD到当前的分支。
这个结果是你所期盼的， 这个远程分支最新的几次提交会被合并进当前的分支中。

git pull

git pull = git fetch + git merge

假如你仅仅想拉取远程分支的代码到本地，还没准备merge到代码中，可以用git fetch,毕竟这样风险比较小，但是git pull也是一种敏捷开发命令，具体使用场景视情况而定。

git fast-forward与no fast-forward的区别

发表于 2016-08-10   |  

Fast-Forward

例如：一直在develop分支上开发，并在其上进行一系列提交，完成时merge master分支，此时，master merge develop分支之后并未产生任何新的commit。此时的合并就叫fast-forward。

Fast-Forward具体定义

当前 master 分支所指向的提交是你当前提交（有关develop的提交）的直接上游，所以 Git 只是简单的将指针向前移动。 换句话说，当你试图合并两个分支时，如果顺着一个分支走下去能够到达另一个分支，那么 Git 在合并两者的时候，只会简单的将指针向前推进（指针右移），因为这种情况下的合并操作没有需要解决的分歧——这就叫做 “快进（fast-forward）”。

使用fast forward

上面可以看出这是一次fast-forward式的合并，且合并后的视图为扁平状，看不出develop的任何提交记录。

禁用fast forward

-—no-ff (no fast foward)，使得每一次的合并都创建一个新的commit记录。即使这个commit只是fast-foward，用来避免丢失信息。

可以看出，使用–no-ff后，会多生成一个commit 记录，并强制保留develop分支的开发记录（而fast-forward的话则是直接合并，看不出之前Branch的任何记录）。这对于以后代码进行分析特别有用。

应用场景

-–no-ff，其作用是：要求git merge即使在fast forward条件下也要产生一个新的merge commit。此处，要求采用–no-ff的方式进行分支合并，其目的在于，希望保持原有“develop branches”整个提交链的完整性。
