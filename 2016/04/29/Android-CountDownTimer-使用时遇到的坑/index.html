CountDownTimer遇到的坑

先来看一段触发CountDownTimer的代码

触发按钮就会触发CountDownTimer计时器，这里我们来分析下CountDownTimer两个参数：

1.millisInFuture 计时器从开始计时到结束计时的周长
2.countDownInterval 计时器计时的间隔时间

我们来分析下CountDownTimer的实现原理:
start

通过start()启动计时器，通过代码可以看出通过mHandler进行一些核心操作，进入到mHandler看看到底干了些什么；

通过代码可以看出实现还是蛮简单的，通过计算delay时间，触发sendMessageDelayed(message, delay)轮训发送消息达到计时的效果。

这里面反复用到SystemClock.elapsedRealtime()计算时间，这里说明下与其他获取时间的区别：
-System.currentTimeMillis() 系统时间，也就是日期时间，可以被系统设置修改，然后值就会发生跳变。
-uptimeMillis 自开机后，经过的时间，不包括深度睡眠的时间
-elapsedRealtime自开机后，经过的时间，包括深度睡眠的时间

从上面的日志里面分析，当onTick返回的时间是1998毫秒时， 整个计时器紧接着就delay了1998毫秒， 然后就直接onFinish了， 计算次数减少一次，且最后一次计数间隔大于一毫秒小于两毫秒。

回到上面场景，我们来分析下到底是什么原因导致的：
当第22行onTick(millisLeft)返回，且millisLeft == 1998毫秒， 继续往下执行，执行到最后一行sendMessageDelayed(obtainMessage(MSG), delay)位置，此时delay为999毫秒， 然后休眠999毫秒之后，重新进入到handleMessage方法，此时13行计算后millisLeft == 997， 进入到16行判断里，millisLeft < mCountdownInterval条件成立， 继续休眠997毫秒， 继续进入handleMessage方法，此时millisLeft == -1毫秒，第14行millisLeft <= 0条件成立，直接调用onFinish()方法，计时结束。

通过分析说明，CountDownTimer计时并不能完全准确，特别是计算一些时间周长短且需求精度高的动作， 所以平时开发时用这个类一定要注意。

万一要用这个类计算建议millisInFuture参数最好不要写成整数，如上例子如果millisInFuture参数写成4012毫秒。
